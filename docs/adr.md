# Architecture Decision Records — elion-dal

Краткие записи ключевых решений, чтобы команда не переспрашивала «почему так».

## ADR-001. Векторная БД — Qdrant

**Контекст.** Нужен гибридный поиск (dense + sparse) с фильтрами по источнику и дате,
на CPU, для небольшого редко меняющегося корпуса.
**Решение.** Qdrant.
**Почему.** Нативная поддержка named-векторов `dense` + `sparse` и RRF-fusion в Query API
из коробки; мощные payload-фильтры; embedded-режим (`:memory:`/on-disk) для dev/CI без
Docker; ANN на CPU (GPU не нужен). Назван в ТЗ.
**Альтернативы.** pgvector (одна БД, но fusion BM25+dense пришлось бы писать руками).

## ADR-002. Гибрид — «вариант A»: BGE-M3 dense + learned sparse

**Контекст.** ТЗ требует «BM25 + dense». BGE-M3 за один прогон отдаёт и dense, и learned
sparse (лексические веса).
**Решение.** Используем dense + learned sparse от BGE-M3 (через FlagEmbedding), fusion = RRF.
**Почему.** Learned sparse лучше классического BM25 на русской морфологии, не требует
отдельного лемматизатора, падает из того же прогона модели. Бэкенд `fastembed`
(multilingual-e5 + BM25) оставлен как лёгкая ONNX-альтернатива для сравнения по латентности.
**Замер (CPU).** BGE-M3: query p50 ≈ 148 ms, индексация ≈ 10 docs/s — приемлемо.
**На будущее.** Третий «настоящий» BM25-сигнал, если всплывут промахи на точном матче
редких строк (номера приказов и т.п.).

## ADR-003. Parent-child retrieval + Postgres как source-of-truth

**Контекст.** Нужен точный поиск и одновременно богатый контекст для генерации; плюс
возможность пересобрать индекс без перекраулинга.
**Решение.** Документ → секции (**родители**) → дочерние чанки. В Qdrant индексируются
**только дети** (с `parent_id` в payload); на поиске гибрид по детям схлопывается в
уникальных родителей, возвращаются секции. Postgres хранит истину (documents/parents/chunks),
Qdrant — производный индекс. Дедуп по `content_hash`; хеш фиксируется только после успешного
upsert в Qdrant (защита от дрейфа при сбое).
**Почему.** Точный матч на маленьких детях + неразорванный контекст из родителя; полная
пересборка индекса из PG; идемпотентность. Контракт с ETL: документ несёт `sections[]`
(желательно из Markdown со структурой), иначе весь документ = один родитель.

## ADR-006. Пивот публичного контракта: REST API вместо gRPC

**Контекст.** После ADR-005 сервер был чисто gRPC. На VibeNest публичный домен идёт
через Caddy/Coolify, который **не пропускает gRPC** (HTTP/2 с `content-type: application/grpc`):
ответ `HTTP/2 :status 501`. Раздельные TCP-порты также не пробрасываются.

**Решение.** Перевести публичный контракт на **REST API (FastAPI)** на том же порту,
где раньше был stdlib /healthz. Все ручки требуют `Authorization: Bearer <token>` (кроме
`/healthz`). Локальная админка — HTTP-клиент (httpx).

**Что НЕ удалили.** Код gRPC сохранён целиком: `proto/`, `servicer.py`, `grpc_gen/`,
`admin/grpc_client.py`, поля `grpc_*` в Settings. В `server.py` запуск gRPC-сервера
закомментирован — раскомментируется, когда платформа научится проксировать gRPC.

**Эквиваленты ручек:** Search → `POST /api/v1/search`; UpsertDocuments →
`POST /api/v1/documents`; DeleteBySource/Doc → `DELETE /api/v1/sources/{id}` /
`.../documents/{id}`; ListSources → `GET /api/v1/sources`; GetStats →
`GET /api/v1/stats`; GetSettings/UpdateSettings → `GET/POST /api/v1/settings`;
HealthCheck → `GET /healthz` (без auth).

**Цена.** Стрим UpsertDocuments стал single-doc POST (для админки достаточно; для
батча — можно добавить bulk-эндпоинт). Локально удобнее тестить (curl), нет gRPC-stub
накладных расходов.

## ADR-005. Сервер только gRPC, админка — отдельный локальный процесс

**Контекст.** Сервис деплоится на платформу с публичным URL. Встроенная HTTP-админка
на том же контейнере → публично доступная панель; Basic-auth не имеет logout, и риск
дыры при недонастроенном `ADMIN_PASSWORD` высок. Платформа build-pack деплоит один
контейнер (compose не поддерживается).

**Решение.** Развести на два процесса:
- **Сервер (деплой)** — только gRPC (50051) + stdlib `/healthz` (8080) для health-проб
  платформы. Без FastAPI/uvicorn, лёгкий образ.
- **Админка (локальная)** — отдельный процесс на машине администратора, тот же
  FastAPI-UI, но под капотом gRPC-клиент к удалённому серверу. Подключение через env
  `GRPC_TARGET` (`host:port`) + `API_TOKEN`. Деп-группа `.[admin]`.

**Почему.** (1) Публично доступна только gRPC-ручка под токеном — атак-поверхность
меньше. (2) Нет проблемы «нельзя выйти» (Basic-auth у админки только локально).
(3) Образ деплоя без FastAPI/uvicorn заметно легче. (4) Один источник правды — gRPC;
любой другой UI/клиент подключается тем же способом.

**Цена.** Загрузка файла идёт через `UpsertDocuments`-стрим (PDF/DOCX парсится на
стороне админки), что требует `.[admin]` локально (loaders уже там). Новые RPC
`GetSettings`/`UpdateSettings` добавлены в контракт.

## ADR-004. Квантизация эмбеддингов — toggle, default OFF

**Контекст.** Нужна экономия RAM на CPU. Добавили флаг `EMBEDDING_QUANTIZE`.
**Решение.** Флаг есть, но **по умолчанию выключен**.
**Почему.** Замер на BGE-M3 (`flag`, torch dynamic int8): RSS процесса ~**4.0 ГБ** с
квантизацией против ~**2.0 ГБ** без неё (`gc.collect()` не помогает). Динамическая
квантизация держит fp32-копию на пике, и аллокатор не возвращает память ОС — то есть для
этой модели int8-toggle **ухудшает** память (польза, если есть, только по скорости CPU).
**Как реально снизить RAM.** int8 **ONNX-экспорт** BGE-M3 (через `optimum`) — отдельная
задача; либо лёгкая ONNX-модель в бэкенде `fastembed`. Toggle оставлен для экспериментов.
